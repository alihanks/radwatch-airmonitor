## Weather parser: missing values as 0.0 vs NaN

**Where:** `image_scripts/weather_utils.py`, `parse_weather_data`.

**Current behavior:** empty cells and non-numeric placeholders (`--`, etc.) are mapped to `0.0`. Historical behavior since before this repo was versioned here.

**Why it matters:** `0.0` and "no reading" are not the same physical statement. A missing solar reading mapped to `0.0` looks like nighttime; a missing rain reading looks like no rain; missing wind speed looks calm. Aggregations and wind roses over these cells carry small biases.

**What a fix would look like:** map missing cells to `math.nan` instead of `0.0`, then audit each consumer downstream (`h5_analysis.py`, `weather_utils.draw_windrose`, rain accumulation logic) to ensure NaN is handled gracefully (e.g. masked from sums, skipped in histograms). Not a one-liner because the plotting code wasn't written with NaN safety in mind across every weather field.

## Historical weather gap: 2017–2024 data should exist somewhere

**Where:** `weatherhawk.csv`.

**Observation:** an earlier weather station was running until 2017 and should have produced historical data. The current CSV has some 2014 entries plus recent 2025+ data, but the 2017–2024 stretch is empty. Since the current RadWatch interest in weather data only begins in 2025, this isn't blocking anything, but recovering the older station's data (if it's recoverable) would be nice for long-term context.

**What a fix would look like:** locate the old station's archive (presumably somewhere on a prior server or backup), decide whether to import it into `weatherhawk.csv` or keep it as a sibling file, and confirm no format changes trip up `parse_weather_data`.

## Pre-2018 spectral data: backwards-compatibility is unverified

**Where:** `current/` only — the pre-2018 CNFs live in the Dropbox `RPAVLOVSKY` folder and are not processed by any existing analysis.

**Observation:** the published pre-2018 numbers were generated by an older pipeline with different analysis procedures (e.g. ROI definitions, calibration, channel count) and live nowhere on the current server. The current pipeline is *intended* to be backwards-compatible — the CNF parser handles the older format — but nobody has actually run older CNFs through end-to-end and compared against the original published numbers.

**What a fix would look like:** pick a known pre-2018 date with a published result, pull those CNFs into a scratch directory, run `raw_analysis.py` + `h5_analysis.py` pointed at that directory, and check whether the numbers reproduce. If yes, document the invocation in the runbook. If no, diagnose the drift (likely calibration-era mismatch).

## Re-evaluate K-40 correction bounds after some time

**Where:** `raw_analysis.py`, `K40_LIVETIME_THRESHOLD = 0.90`, `K40_LIVETIME_MIN_RATIO = 0.70`.

**Context:** the p75 baseline + 0.70 raw-drop threshold landed on 2026-04-21 and appear to resolve the observed edge dips. These thresholds were picked from first principles, not tuned against a sample of known-good vs known-bad spectra.

**What a revisit would look like:** after a few weeks of accumulated data, check whether hourly QA (`qa_flagged.csv`) is still firing meaningfully. If it's essentially silent, the raw-spectrum drop is doing the work and the hourly QA might be simplified or removed. If it's firing a lot, the raw-drop threshold may be too lenient — either raise `K40_LIVETIME_MIN_RATIO` or tighten `K40_MIN_RATE`/`QA_THRESHOLD`.
