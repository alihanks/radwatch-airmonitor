# RadWatch Air Monitor — Operations Runbook

Practical reference for running specific tools in isolation: rebuilding the HDF5, filling weather gaps, diagnosing stale plots, deploying manually, and reading QA output. All paths below assume the pipeline's server install at `/home/dosenet/radwatch-airmonitor/` and the `radwatch` conda env.

```bash
ssh dosenet
cd ~/radwatch-airmonitor
conda activate radwatch
```

---

## 1. Full rebuild of `rebin.h5`

**When:** After a change that invalidates the stored HDF5 contents — e.g. HDF5 schema change, channel-count standardization change, K-40 baseline change (like the 2026-04-20 p75 fix), calibration update, or suspected corruption. Any change noted in the logbook with "delete `rebin.h5` and `last_processed.txt`" as a deploy note is this scenario.

**Scope:** The rebuild reads all CNFs under `/home/dosenet/Dropbox/UCB Air Monitor/Data/Roof/current/`. That's the **2025 and 2026 data only**. See *Historical data* below.

**Steps:**

```bash
# 1. (Optional) Back up the current HDF5 if you want to compare before/after.
cp data/rebin.h5 data/rebin.h5.prerebuild

# 2. Delete the state that gates the incremental path.
rm data/rebin.h5 data/last_processed.txt

# 3a. Let the next cron run rebuild, OR
# 3b. Trigger manually (from project root):
python3 image_scripts/analysis/raw_analysis.py
python3 image_scripts/analysis/h5_analysis.py

# 4. Check progress as it runs.
tail -f data/pipeline.log
```

**What to look for in the log:**
- `K-40 gross counts baseline: median=X, p75=Y (using p75)` — appears once during the raw-analysis pass; confirms the livetime-correction baseline. Expect Y near ~100 on clean data; Y/X ratio of ~1.0–1.4 is the bimodality bias.
- `K40 livetime correction: <timestamp> | original=...s → corrected=...s | K40 ratio=0.XXX` — one line per raw spectrum with K-40 ratio in `[0.70, 0.90)`; livetime scaled down proportionally.
- `K40 spectrum drop: <timestamp> | K40 ratio=0.XXX < min 0.7 (not a reliable livetime correction)` — one line per raw spectrum with ratio below 0.70; spectrum is dropped before rebinning.
- `estimate_livetimes_from_k40: corrected N, dropped M, of K spectra` — per-run summary across the raw-spectrum K-40 pass.
- `Dropped N raw spectra below K40 ratio 0.7; M raw spectra remain for rebinning` — reports the filtered collection size just before rebin.
- `Rebinning data...` — once per hour-bin pass, fast.
- `QA flagged N anomalous samples` — hourly-level QA flagged-sample count; distinct from raw-spectrum drops above. Details in `data/qa_flagged.csv`.
- `Database written` — final HDF5 write. If you don't see this, the rebuild didn't complete.

**Expected runtime:** Proportional to the size of `current/`. Reading CNFs is the bottleneck; a full rebuild over a year's worth of hourly spectra takes several minutes of I/O.

**Historical data (pre-2018) — do not rebuild:**
An earlier version of the pipeline pointed at the `RPAVLOVSKY` folder in Dropbox and used different analysis procedures. That pre-2018 data should **not** be fed through the current pipeline — results would not match what was previously published. The new analysis procedure is intended to be backwards-compatible in the sense that someone who wanted to recover those earlier results *could* run them through (the parsing code handles the older CNFs), but the published pre-2018 numbers live nowhere on the current server and intentionally aren't regenerated by the current `current/`-only rebuild.

## 2. Weather backfill

The weather pipeline has two distinct modes. The cron job runs only the first.

### 2a. Trailing catch-up (default, automatic)

**What it does:** Reads the last date in `weatherhawk.csv`, scrapes every day from there to today from Weather Underground, appends rows.

**When it runs:** Every cron invocation, automatically, as the first step in `cron_job.sh`.

**Manual invocation** (useful if cron didn't run and you want to catch up now without going through the full pipeline):

```bash
python3 image_scripts/weather_gatherer.py
```

### 2b. Internal gap fill (manual only)

**What it does:** Scans for dates *between* the first and last entries in `weatherhawk.csv` where data is missing (e.g. days the scraper wasn't running) and backfills them from WU. Uses a 2-second-per-request delay.

**When to run:** When you see gaps in the middle of a weather series in the plots, or when `weatherhawk.csv` has known holes from downtime. Not needed for trailing catch-up (that's 2a).

**Always use `--since`** to cap the scan start; otherwise it will try every day back to the CSV's oldest entry (which may be a stale pre-2018 row):

```bash
python3 image_scripts/weather_gatherer.py --fill-gaps --since 2025-01-01
```

The log prints:
```
CSV date range: <first> to <last>
Scan range:     <since or first> to <last>
```
Verify the Scan range is sensible before it starts scraping.

### 2c. Re-sort after backfill

Backfilled rows append out of chronological order. The main pipeline re-sorts automatically via `resort_weather_timestamps()` in `raw_analysis.py`, so the fix happens on the next pipeline run. To re-sort immediately without a pipeline run:

```bash
python3 -c "from image_scripts.weather_utils import resort_weather_timestamps; \
  resort_weather_timestamps('weatherhawk.csv', 'data/weather_sorted.csv')"
```

## 3. K-40 baseline diagnostic

**What it does:** Reads a recent window of raw CNFs from `current/`, computes K-40 gross counts per raw spectrum, and prints summary stats (median, p75, p90) plus a text histogram. Read-only; touches no state.

**When to run:**
- Before a full rebuild, to confirm the distribution looks bimodal (p75 meaningfully above median → livetime correction will land on the right baseline).
- After a detector or firmware change, to check whether typical K-40 counts per spectrum have shifted.
- Any time the K-40 count rate in the plots looks off and you want a quick distributional view of the raw data.

**Command:**

```bash
python3 image_scripts/analysis/diagnose_k40_baseline.py            # default 500 files
python3 image_scripts/analysis/diagnose_k40_baseline.py --n 2000   # larger sample
```

**How to read the output:**
- `median ≈ p75` (ratio within ~5%) → distribution is unimodal; all spectra are roughly full-livetime. Nothing to fix.
- `p75 ≈ 1.1–1.4 × median` → bimodal (full-livetime + truncated); the pipeline's p75-based baseline correction is appropriate and is working on the right target.
- `p75 ≫ 1.5 × median` → unusually heavy truncated tail. Investigate whether the detector is stopping collections early more than usual (MCA issue, power instability).
- Very low `max` value (well below ~100) → either calibration has drifted or the detector is producing much fewer counts than nominal. Investigate.

**Related pipeline behavior (set in `raw_analysis.py`):** raw spectra are compared to the stored p75 baseline. Ratio ≥ `K40_LIVETIME_THRESHOLD` (0.90) → no correction. Ratio in `[K40_LIVETIME_MIN_RATIO, K40_LIVETIME_THRESHOLD)` (`[0.70, 0.90)`) → livetime scaled down proportionally. Ratio < `K40_LIVETIME_MIN_RATIO` (0.70) → raw spectrum is **dropped** before rebinning, because the linear correction isn't reliable at that depth of K-40 deficit (spectrum likely has problems beyond short livetime). The drop mechanism logs per-sample and is separate from the hourly QA filter below (see §6).

## 4. Diagnosing stale plots on the web

Plots on the RadWatch website are produced by the hourly cron job. "Stale" = the timestamps on the isotope plots aren't advancing, or the web dashboard shows the same data day after day.

**Walk the pipeline, stopping at the first failure:**

```bash
# 1. Is the cron job actually firing?
tail -200 data/pipeline.log        # most recent run log
grep "Pipeline run:" data/pipeline.log | tail -5   # last 5 run timestamps

# 2. Is the spectral data arriving via Dropbox?
ls -d '/home/dosenet/Dropbox/UCB Air Monitor/Data/Roof/current/'*/ | tail -5
# Most-recent directory's mtime should be within the last few hours.

# 3. Is the incremental pipeline picking up new files?
cat data/last_processed.txt
# If this points to a file in a directory that no longer exists in Dropbox,
# delete it so the next run starts fresh:
#   rm data/last_processed.txt

# 4. Is the HDF5 being written?
ls -lh data/rebin.h5
# mtime should be within the last hour on a healthy system.

# 5. Are the PNGs being regenerated?
ls -lh data/*.png
# Each of iso_*.png, rose_*.png, trimmed_*.png, out.png, etc. should have
# the same recent mtime.

# 6. Is the deploy step working?
grep "Deploying\|Deploy exit code" data/pipeline.log | tail -10
# Exit code 0 = success. Nonzero = see section 5.
```

**Most common causes, top down:**
- Cron not running at all (user changed, conda path broken, host rebooted without re-adding env).
- Dropbox desktop daemon stopped syncing (no new directories in `current/`).
- Stale `last_processed.txt` pointing to a removed directory (pipeline sees "nothing new").
- SFTP deploy failing (plots are fresh on server but not uploaded).

## 5. Manual deploy / SFTP failure

**Symptom:** Pipeline log shows `Deploy exit code: <nonzero>` or `ERROR: RADWATCH_SFTP_PASS not set`.

**Check the env var is set:**

```bash
echo "SFTP pass set: $([ -n "$RADWATCH_SFTP_PASS" ] && echo yes || echo no)"
```

**Persistent places it should be set:**
- In the crontab line before the cron_job.sh call: `RADWATCH_SFTP_PASS=... 0 * * * * ...`
- Or in `~/.bashrc` (only picked up by interactive shells, not cron — cron needs the env var in the crontab itself).

**Manual deploy** (after a successful pipeline run, or after rebuilding `rooftop_tmp/` contents):

```bash
export RADWATCH_SFTP_PASS='...'                 # only if not already set
bash image_scripts/analysis/deploy.sh
```

The script uses `lftp` with an `EOF` heredoc mirror. If it hangs or fails mid-upload, the lftp output will show which file failed. Target is `sftp://coeradwatch.sftp.wpengine.com:2222/test/` under user `coeradwatch-RADWATCH`.

## 6. Sample-level filtering (raw drops + hourly QA)

Two filters operate at different levels; both are worth knowing about when diagnosing missing data.

### 6a. Raw-spectrum drops (pre-rebin)

Applied in `raw_analysis.py` during the K-40 livetime-correction pass (see §3). When a raw spectrum's K-40 counts are below `K40_LIVETIME_MIN_RATIO × baseline` (currently `0.70 × p75`), the spectrum is removed from the collection before rebinning. These are **not** recorded in `qa_flagged.csv` — they only appear in `pipeline.log` as `K40 spectrum drop: <timestamp> | K40 ratio=0.XXX`.

Count them via:
```bash
grep -c "K40 spectrum drop:" data/pipeline.log
```

Dropping at the raw level is surgical: only that one ~300s spectrum is removed, and the other spectra contributing to the same hourly bin still contribute to all 7 isotope rates.

### 6b. Hourly QA flagged samples (post-rebin)

**File:** `data/qa_flagged.csv`.

**Columns:** `timestamp, k40_rate, rolling_median, reason`.

**Reason strings (from `raw_analysis.py`):**
- `K40 rate X.XXXX < absolute minimum 0.15` — hourly K-40 rate fell below the hard floor. Hard floor exists to catch obviously bad data even when the rolling median hasn't stabilized (e.g. start of a fresh rebuild). Controlled by `K40_MIN_RATE` in `raw_analysis.py`.
- `K40 rate X.XXXX < 50% of median Y.YYYY` — K-40 dropped to under half of the 24-hour rolling median. Catches sudden mid-series drops that aren't low in absolute terms. Controlled by `QA_THRESHOLD` (0.5) and `QA_WINDOW` (24) in `raw_analysis.py`.

When the raw-spectrum drop (6a) is catching the problem cleanly, hourly QA should rarely trigger. If hourly QA is firing often, the raw-spectrum drop threshold may need revisiting (see `docs/todo.md`).

**Looking for patterns:**

```bash
# Count of flagged samples per day
awk -F, 'NR>1 {split($1,a," "); print a[1]}' data/qa_flagged.csv | sort | uniq -c | tail -20

# Latest 20 flagged samples
tail -20 data/qa_flagged.csv
```

**Interpretation:**
- Occasional single-sample flags: normal detector hiccups, nothing to do.
- Multi-hour clusters of flags: likely a real detector issue (electronics, power, shielding disturbance) — check the raw CNFs in the corresponding date directory.
- Flags that all say "absolute minimum" during the first day after a rebuild: expected (rolling median baseline hadn't stabilized yet); those samples are dropped but everything after steady-state is fine.

The QA filter does not affect the source CNFs in Dropbox; it only decides which rebinned hourly samples to retain in `rebin.h5`.
